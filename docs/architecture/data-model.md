# 데이터 모델 (PostgreSQL)

Status: **Ready for Implementation (v2 MVP)**  
Owner: everbit  
Last updated: 2026-03-11 (Asia/Seoul)

이 문서는 “정합성/멱등/상태 머신”을 DB 레벨에서 보장하기 위한 **최소 스키마**를 고정한다.  
구현은 JPA를 사용하더라도, **유니크 제약과 상태 전이, 물리 FK/논리 관계의 기준은 이 문서가 최종 SoT** 이다.

- ERD는 관계를 빠르게 검증하기 위한 **시각화 도구**이며, 스펙은 본 문서의 제약/상태 정의가 기준이다.
- 실행 가능한 DDL 초안은 `docs/db/schema-v2-mvp.sql` 를 따른다.
- JPA 구현 시 복합키/공유PK/복합 FK 매핑 표준은 `docs/architecture/jpa-mapping.md` 를 따른다.
- **테이블/컬럼 네이밍**: Spring Boot `SpringPhysicalNamingStrategy`가 camelCase → snake_case 자동 매핑하므로, JPA 엔티티에서 `@Table`/`@Column` name을 명시할 필요가 없다.

## 0. ERD (Mermaid)

> 주의 1: `outbox_event`는 `aggregate_type + aggregate_id`의 **폴리모픽 참조**이므로 ERD에서 FK로 연결하지 않는다.  
> 주의 2: `signal.strategy_key`, `signal.market`는 **히스토리 스냅샷 값**이며, 변경 가능한 설정 테이블과는 의도적으로 물리 FK를 두지 않는다.

## 1. 공통 규칙

### 1.1 키/시간

#### 1) PK(내부 조인/성능 기준)
- 기본 PK는 `bigint GENERATED BY DEFAULT AS IDENTITY` 를 사용한다.
- 예외(의도적):
  - **1:1 테이블**은 FK를 곧 PK로 사용한다(공유 PK).
    - `upbit_key.owner_id` = PK = FK(`app_user.id`)
    - `kill_switch.owner_id` = PK = FK(`app_user.id`)
    - `backtest_result.job_id` = PK = FK(`backtest_job.id`)
  - **자연키가 명확하고 설정/상태 성격이 강한 테이블**은 복합 PK를 사용한다.
    - `strategy_config (owner_id, strategy_key)`
    - `market_config (owner_id, market)`
    - `market_state (owner_id, market)`
    - `position (owner_id, market)`
    - `candle_cache (market, timeframe, candle_time)`

#### 2) 외부 노출/추적용 ID(UUID v7)
- 내부 `id(bigint)`는 **외부 API/URL/로그에 노출하지 않는다.**
- 외부 노출 및 상관관계 추적용으로 `public_id uuid`(UUID v7)을 둔다.
  - 표준: RFC 4122 UUID v7 (시간순 정렬 가능)
  - 생성 주체: 애플리케이션(서버)
  - 제약: `UNIQUE(public_id)` + `NOT NULL`

> v2 MVP 기준 `public_id` 적용 테이블: `app_user`, `signal`, `order_intent`, `order_attempt`, `backtest_job`  
> (필요 시 P1+에서 확대)

#### 3) UUID 사용(외부 시스템/이벤트 식별)
- Upbit 주문/체결 식별자는 **UUID 형태**이므로 PostgreSQL `uuid` 타입을 사용한다.
  - `upbit_order.upbit_uuid`
  - `order_attempt.upbit_uuid`
  - `fill.trade_uuid`
- Outbox 이벤트의 dedupe/추적 키는 `uuid` 타입 `event_id`를 사용한다.

#### 4) 공통 감사 컬럼(BaseEntity)
- JPA 엔티티는 원칙적으로 **`BaseEntity(created_at, updated_at)`** 를 상속한다.
- 따라서 **엔티티 테이블에는 `created_at`, `updated_at` 두 컬럼을 모두 둔다.**
  - append-only 성격의 row라도 `updated_at`은 유지한다. 일반적으로 insert 직후 `created_at == updated_at` 이다.
- 비즈니스 시각은 별도 컬럼으로 둔다.
  - 예: `candle_close_time`, `trade_time`, `captured_at`, `rotated_at`, `suspended_at`, `delivered_at`
- 시간은 `timestamptz`(UTC)로 저장하고, 표시만 KST로 변환한다.
- Java 타입은 `Instant`를 권장한다. `LocalDateTime`을 쓸 경우에는 **애플리케이션/JDBC/Hibernate timezone을 UTC로 고정**해야 한다.

### 1.2 JSON 사용
- 전략/요청 파라미터는 `jsonb`로 저장한다(스키마 버전 필수).

### 1.3 상태/이벤트
- 상태 전이는 “이전 상태 → 다음 상태”가 명시되어야 한다(임의 업데이트 금지).
- 도메인 이벤트/커맨드 발행은 **Outbox/Queue(`outbox_event`)** 로 수행한다.

### 1.4 숫자 타입(정밀도)
- 거래 수치(가격/수량/수수료/손익 등)는 `numeric(38,18)`을 기본으로 사용한다.
  - v2 MVP의 데이터 규모에서는 성능보다 정합성이 우선이다.
  - 단, `strength` 같은 스코어링 값은 `numeric(18,8)` 등 축소 가능.
- 명확히 음수가 될 수 없는 값은 CHECK 제약으로 방어한다.
  - 예: 주문 수량/금액, 체결 수량, 평균단가, 캔들 OHLC

### 1.5 상태 값(허용 리스트)
DB는 문자열 컬럼을 쓰되, v2 MVP에서는 **CHECK 제약으로 오타/불법 상태를 차단**한다.

- `app_user.role`: `ADMIN`, `USER`
- `app_user.everbit_key_status`: `NONE`, `PENDING`, `ACTIVE`, `REVOKED`
- `everbit_key_request.status`: `PENDING`, `APPROVED`, `REJECTED`
- `everbit_key.status`: `ACTIVE`, `REVOKED`
- `signal.side`: `BUY`, `SELL`
- `order_intent.intent_type`: `ENTRY`, `EXIT_STOPLOSS`, `EXIT_TP`, `EXIT_TRAIL`, `EXIT_TIME`
- `order_intent.status`: `CREATED`, `CANCELED`, `COMPLETED`
- `order_attempt.status`: `PREPARED`, `SENT`, `ACKED`, `REJECTED`, `THROTTLED`, `UNKNOWN`, `SUSPENDED`
- `position.status`: `FLAT`, `OPEN`
- `market_state.trade_status`: `ACTIVE`, `SUSPENDED`
- `outbox_event.status`: `PENDING`, `PROCESSING`, `DONE`, `DEAD`
- `backtest_job.status`: `QUEUED`, `RUNNING`, `DONE`, `FAILED`

### 1.6 물리 관계 vs 논리 관계(중요)
- **물리 FK를 두는 관계**
  - 생명주기가 명확한 부모-자식 관계
  - owner 일치가 반드시 보장되어야 하는 관계
  - 예: `order_intent → signal`, `order_attempt → order_intent`, `fill → upbit_order`
- **논리 관계로만 유지하는 관계**
  - 과거 히스토리 row가 현재 설정 row의 존재 여부에 종속되면 안 되는 경우
  - 예: `signal.strategy_key ↔ strategy_config`, `signal.market ↔ market_config`
- `owner_id`를 중복 보유하는 테이블(`signal`, `order_intent`, `order_attempt`, `upbit_order`, `fill`)은 조회/인덱싱 편의를 위해 유지하되,
  **복합 FK/UNIQUE로 부모와 같은 owner임을 강제**한다.
- `outbox_event`는 폴리모픽이므로 물리 FK를 두지 않는다.

## 2. 핵심 테이블(트레이딩)

> 네이밍: PostgreSQL에서 `user`는 혼란을 유발할 수 있으므로 테이블명을 `app_user`로 고정한다.

### 2.1 app_user (멀티유저 — ADMIN / USER)

> ADR: [ADR-0010](../../adr/0010-multi-user-everbit-key.md)

| Column | Type | Notes |
|---|---|---|
| owner_id | bigint PK(identity) | 내부 PK (사용자 식별자) |
| public_id | uuid unique | 외부 노출 UUID v7 |
| kakao_id | varchar unique | 카카오 계정 ID |
| email | varchar nullable | |
| role | varchar | ADMIN / USER |
| everbit_key_status | varchar | NONE / PENDING / ACTIVE / REVOKED (ADMIN은 항상 NONE으로 간주, 검사 생략) |
| created_at | timestamptz | 생성 시각 |
| updated_at | timestamptz | 마지막 수정 시각 |

제약:
- UNIQUE(kakao_id)
- UNIQUE(public_id)
- CHECK role IN ('ADMIN', 'USER')
- CHECK everbit_key_status IN ('NONE', 'PENDING', 'ACTIVE', 'REVOKED')

비즈니스 규칙:
- 최초 가입자는 role = `ADMIN`으로 등록된다. 이후 가입자는 `USER`로 등록된다.
- `USER`의 실거래 게이트는 `everbit_key_status = ACTIVE`를 요구한다.
- `ADMIN`은 `everbit_key_status` 조건을 건너뛴다.

- `(order_intent_id, owner_id)`는 부모 OrderIntent와 owner 일치가 보장되어야 함

역할:
- `order_intent`는 semantic intent
- `order_attempt.request_json`은 executable snapshot
- outbox `CreateOrderAttempt` payload는 **orderAttemptId만** 전달

## 3. Outbox/Queue(필수)

v2 MVP에서는 Kafka 없이 PostgreSQL을 이벤트 버스/큐로 사용한다.  
도메인 변경(SoT)과 “비동기 처리 트리거”를 동일 트랜잭션으로 묶기 위해 `outbox_event` 테이블을 사용한다.

- 발행: 트랜잭션 내 `outbox_event` INSERT
- 소비: 워커가 `FOR UPDATE SKIP LOCKED`로 claim 후 처리
- 재시도/백오프: `attempt_count`, `next_retry_at` 등으로 DB에서 관리

상세 스펙: `docs/architecture/event-bus.md`

### 3.1 outbox_event
| Column | Type | Notes |
|---|---|---|
| outbox_event_id | bigint PK(identity) | 처리 순서/인덱싱 |
| event_id | uuid unique | dedupe/추적 키 |
| stream | varchar | 예: everbit.trade.command |
| event_type | varchar | CreateOrderAttempt/OrderAccepted/... |
| aggregate_type | varchar | 예: ORDER_INTENT, ORDER_ATTEMPT |
| aggregate_id | bigint nullable | v2 MVP 기준 bigint aggregate만 사용 |
| payload_json | jsonb | envelope 또는 payload |
| status | varchar | PENDING/PROCESSING/DONE/DEAD |
| attempt_count | int | default 0 |
| max_attempts | int | default 10(권장) |
| next_retry_at | timestamptz | default now() |
| last_attempt_at | timestamptz nullable | |
| locked_by | varchar nullable | 워커 식별자 |
| locked_until | timestamptz nullable | 스테일 락 복구용 TTL |
| last_error_code | varchar nullable | |
| last_error_message | text nullable | |
| created_at | timestamptz | 생성 시각 |
| updated_at | timestamptz | 마지막 수정 시각 |
| processed_at | timestamptz nullable | DONE 시각 |

인덱스(권장):
- INDEX(stream, status, next_retry_at)
- INDEX(status, next_retry_at)
- INDEX(created_at)

제약:
- UNIQUE(event_id)

주의:
- 외부 호출(Upbit/WebPush)은 **DB 락을 잡은 채로 수행하지 않는다**.
- claim(점유) 트랜잭션은 짧게 종료하고, 처리 결과만 업데이트한다.
- (P1+) aggregate_id가 bigint가 아닌 이벤트가 필요해지면 `payload_json`에 `aggregate_key`를 추가한다.

## 6. 마이그레이션/스키마 변경 규칙

- 스키마 변경은 반드시 ADR 또는 변경 로그를 남긴다.
- 유니크/인덱스/복합 FK는 “성능 최적화”가 아니라 “정합성 스펙”이므로 임의 변경 금지.
- `BaseEntity(created_at, updated_at)`는 엔티티 표준이므로 신규 엔티티도 동일 규칙을 따른다.
- 실행 가능한 DDL 초안: `docs/db/schema-v2-mvp.sql` (v2 MVP)
