# ADR-0001: Git 브랜치/커밋 전략 (v1-v2)

Status: **Accepted**  
Date: 2026-02-14

---

## Context
everbit 레포지토리는 v1 작업이 `main` 브랜치에 누적되어 있다.  
v2는 기존 레포를 유지하면서 v1/v2 이력을 모두 관리해야 하며, 향후 v1 핫픽스 가능성도 있다.

또한 v2는 “문서 우선” 개발을 고정해야 한다.

---

## Decision

### 브랜치
| 브랜치 | 용도 | 비고 |
|---|---|---|
| `v1` | v1 안정 버전 고정 | 핫픽스만 |
| `v2` | v2 개발 | 모든 개발은 여기서 |
| `main` | 선택: 최신 안정 | 필요 시 v1/v2 머지 |

규칙:
- `main`의 현 상태를 v1 스냅샷으로 간주하고 `v1.0.0` 태그 생성
- `v1` 브랜치 생성(또는 유지) 후 v1 유지보수는 `v1`에서만 수행
- v2 개발은 `v2`에서만 수행
- v2 안정화 후 `main <- v2` 병합 + `v2.0.0` 태그 생성

### 커밋(문서 우선)
- 새 기능/변경 사항은 **구현 전에** 관련 문서(FRD/Architecture/ADR/Strategy)를 먼저 커밋한다.
- 문서가 없으면 구현 PR/머지 전에 문서가 먼저 보완되어야 한다.

커밋 메시지:
- `docs: ...`, `feat: ...`, `fix: ...`, `test: ...` 등 prefix 사용

---

## Consequences
- v1/v2 이력과 릴리즈 기준이 분리된다.
- v2 개발 중에도 v1 핫픽스가 가능하다.
- 문서 없이 구현이 먼저 진행되는 것을 방지한다.

---

## Alternatives
- v2를 새 레포로 분리(이력 분산/운영 분리 비용)
- `main`만 사용하고 v1은 태그로만 관리(v1 핫픽스 흐름 불명확)
